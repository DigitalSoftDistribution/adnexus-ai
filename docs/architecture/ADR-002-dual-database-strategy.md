# ADR-002: Dual Database Strategy and Consolidation Path

**Date:** 2026-05-30
**Status:** Accepted
**Scope:** `apps/api/src/`

## Context

The codebase uses two database access patterns simultaneously:

1. **Supabase JS Client** (`src/lib/supabase.ts`) — Used by all legacy v1 routes in `src/routes/`. Provides ORM-like query building, real-time subscriptions, and auth integration.
2. **Raw pg Pool** (`src/infrastructure/database/connection.ts`) — Used by all Clean Architecture repositories in `src/infrastructure/repositories/`. Direct SQL with parameterized queries.

## Current State

### Supabase JS (legacy v1 routes)

```typescript
// Pattern found in src/routes/*.ts
const supabase = getSupabaseClient();
const { data, error } = await supabase
  .from('campaigns')
  .select('*')
  .eq('workspace_id', workspaceId);
```

Used in 24 legacy route files totaling 13,662 lines. The Supabase client is initialized in `src/lib/supabase.ts` and provides:
- Query builder (`from().select().eq()`)
- Row Level Security context (via auth headers)
- Real-time subscriptions
- Storage API
- Auth API (delegated to Supabase Auth)

### pg Pool (Clean Architecture v2)

```typescript
// Pattern in src/infrastructure/repositories/*.ts
const { rows } = await query<Campaign>(
  `SELECT * FROM campaigns WHERE id = $1 LIMIT 1`,
  [id],
);
```

Used by 13 repository implementations. The pool is configured in `src/infrastructure/database/connection.ts` with:
- Connection pooling (max 20 connections)
- Slow query logging (>500ms threshold)
- Transaction support via `transaction()` helper
- Direct access to all PostgreSQL features

## Decision

Standardize on **raw pg Pool** as the canonical database client for all data access in `apps/api/`.

### Rationale

1. **Type safety**: Repository methods map SQL results to typed domain entities with explicit column mapping
2. **Performance**: No ORM overhead; full control over query shape, JOINs, and aggregation
3. **SQL capability**: Direct access to PostgreSQL-specific features (CTEs, window functions, `FILTER`, `jsonb_object_agg`)
4. **Testability**: Repositories can be tested against real PostgreSQL without Supabase client mocking
5. **Consistency**: One query pattern across the entire codebase
6. **Operational clarity**: Queries are visible in code, not generated by an ORM

### What stays with Supabase

Supabase JS remains for:
- **Auth**: `supabase.auth.signIn()`, `supabase.auth.getUser()`, token refresh — the Supabase Auth service handles JWT verification
- **Storage**: File uploads via `supabase.storage` (until replaced by `IFileStorageService`)
- **Realtime**: WebSocket subscriptions for live updates

These use Supabase as a *service*, not as a database client.

## Migration Guide

### Pattern: Simple SELECT

```typescript
// BEFORE (Supabase JS)
const { data } = await supabase
  .from('campaigns')
  .select('*')
  .eq('workspace_id', workspaceId)
  .eq('status', 'active');

// AFTER (pg Pool)
const { rows } = await query<Campaign>(
  `SELECT * FROM campaigns WHERE workspace_id = $1 AND status = $2`,
  [workspaceId, 'active'],
);
return rows;
```

### Pattern: INSERT with RETURNING

```typescript
// BEFORE
const { data, error } = await supabase
  .from('campaigns')
  .insert({ workspace_id: wid, name: 'Test' })
  .select()
  .single();

// AFTER
const { rows } = await query<Campaign>(
  `INSERT INTO campaigns (workspace_id, name) VALUES ($1, $2) RETURNING *`,
  [wid, 'Test'],
);
return rows[0];
```

### Pattern: UPDATE partial

```typescript
// BEFORE
const { data } = await supabase
  .from('campaigns')
  .update({ name: newName, status: 'paused' })
  .eq('id', id)
  .select()
  .single();

// AFTER
const { rows } = await query<Campaign>(
  `UPDATE campaigns SET name = $1, status = $2 WHERE id = $3 RETURNING *`,
  [newName, 'paused', id],
);
return rows[0];
```

### Pattern: Dynamic filters with pagination

```typescript
// BEFORE
let q = supabase.from('campaigns').select('*', { count: 'exact' });
if (status) q = q.eq('status', status);
if (search) q = q.ilike('name', `%${search}%`);
q = q.range(offset, offset + limit - 1);

// AFTER — see CampaignRepository.list() for full example
const conditions: string[] = ['workspace_id = $1'];
const params: unknown[] = [workspaceId];
let idx = 1;
if (status) { conditions.push(`status = $${++idx}`); params.push(status); }
if (search) { conditions.push(`name ILIKE $${++idx}`); params.push(`%${search}%`); }
const whereClause = conditions.join(' AND ');
const { rows } = await query(`SELECT * FROM campaigns WHERE ${whereClause} LIMIT $${++idx} OFFSET $${++idx}`,
  [...params, limit, offset],
);
```

### Column naming convention

Database uses `snake_case` (e.g., `workspace_id`, `created_at`). Domain entities use `camelCase` (e.g., `workspaceId`, `createdAt`). Repository implementations handle the mapping via `camelToSnake()` helper and explicit column aliases in SELECT queries.

## Deprecation Timeline

| Phase | Action |
|-------|--------|
| Current | New code uses pg Pool exclusively. Legacy routes unchanged. |
| Migration | Each legacy route migrated to Clean Architecture uses pg Pool |
| Final | Remove `src/lib/supabase.ts` data-access imports (keep auth/storage) |

## Consequences

**Positive:**
- Single query pattern for all data access
- Full PostgreSQL feature access
- Better query performance visibility (slow query logging)
- Easier to write integration tests

**Negative:**
- More verbose SQL in repository files vs ORM query building
- Manual column mapping between snake_case DB and camelCase entities
- Supabase RLS policies bypassed by direct pg Pool (must handle authorization in application layer)
